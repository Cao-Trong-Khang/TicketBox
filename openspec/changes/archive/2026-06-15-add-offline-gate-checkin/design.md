## Context

TicketBox already defines a native Android Check-in Mobile App using Room Database and Jetpack Compose, a NestJS modular-monolith Backend API, PostgreSQL as the authoritative database, Redis for rate limiting and temporary coordination, and Kafka for asynchronous workflows. The global design includes `tickets`, `vip_guests`, `check_ins`, RBAC permissions for Check-in Staff, and an offline check-in protection model where local scan logs are synchronized later.

This change turns that blueprint into a concrete feature design for venue gate operations. The primary stakeholder is Check-in Staff assigned to a concert or gate. Audience users are affected through e-ticket validation. Organizers are affected operationally, but this change does not add organizer reporting or configuration screens.

## Goals / Non-Goals

**Goals:**

- Let assigned Check-in Staff preload validation data for their assigned concert or gate.
- Validate QR-code e-tickets and VIP guest-list entries quickly on Android when online or offline.
- Persist every scan attempt in durable local mobile storage before showing a final local result.
- Synchronize pending local scan logs to the Backend API when connectivity returns.
- Resolve duplicates and conflicts in PostgreSQL so only one successful check-in exists per ticket or VIP guest.
- Return per-scan sync outcomes that the mobile app can display and keep for auditability.
- Keep the feature within the existing Client-Server, Layered, and Event-driven architecture.

**Non-Goals:**

- No new external systems, cloud services, databases, or message brokers.
- No real-time sponsor API, webhook, or direct sponsor database integration.
- No seat-level booking, payment settlement, or ticket issuance changes.
- No organizer-facing analytics expansion beyond data produced by the check-in workflow.
- No guarantee that offline validation can know about scans performed on another device until synchronization.

## Decisions

### Decision 1: Use Android Room as the offline source for assigned event snapshots and scan logs

The mobile app will store assigned event metadata, ticket validation records, VIP guest validation records, staff assignment metadata, and local scan logs in Room. A scan log includes a generated `local_scan_id`, scanned payload hash, entity type, local validation result, staff user ID, source device ID, scanned timestamp, sync status, retry count, and the latest backend sync result.

**Rationale:** This aligns with the global native Android direction and provides durable local storage that survives app restarts and network loss.

**Alternatives considered:** Keeping scans only in memory would lose records during app crashes. Using a new remote service or database would violate the global architecture and fail under connectivity loss.

### Decision 2: Preload only assigned concert or gate data

The Backend API will expose authenticated check-in preload endpoints that return only tickets, VIP guests, event metadata, and assignment data for concerts or gates assigned to the staff user. The mobile app will refresh this data before an event and opportunistically while online.

**Rationale:** Assignment-scoped preload reduces data exposure on mobile devices and matches the existing RBAC model where Check-in Staff can scan only assigned concerts or gates.

**Alternatives considered:** Loading all event tickets to every staff device would simplify offline lookup but would overexpose sensitive ticket and guest data.

### Decision 3: Make PostgreSQL the final authority for sync outcomes

The Backend API will accept batches of pending scan logs and process each scan inside server-side authorization and domain validation. PostgreSQL will enforce at most one successful `check_ins` row per `ticket_id` and at most one successful row per `vip_guest_id`. Later scans for the same ticket or guest become duplicate or conflict outcomes instead of successful check-ins.

**Rationale:** The global design requires PostgreSQL to determine final check-in validity. Backend-first conflict resolution handles multiple offline devices scanning the same ticket independently.

**Alternatives considered:** Trusting the first mobile device to mark a ticket as used would fail across multiple offline devices. Using Redis as the final duplicate guard would contradict the global constraint that Redis is not authoritative.

### Decision 4: Use idempotent batch synchronization with local scan IDs

Each mobile scan will carry a stable `local_scan_id` generated by the app and a `source_device_id`. The backend will store enough metadata to make repeated uploads of the same scan idempotent for the same device. Retrying a sync batch must return the existing outcome instead of creating another check-in or conflict record.

**Rationale:** Mobile networks often fail mid-request. Idempotent sync lets WorkManager retry safely without double-recording scans.

**Alternatives considered:** Accepting anonymous scan rows without local identifiers would make retries ambiguous and could inflate duplicate records.

### Decision 5: Keep Kafka optional for the synchronous check-in path

The sync API will write authoritative check-in and audit records synchronously in PostgreSQL. It may publish future analytics or projection events to Kafka after successful persistence, but Kafka availability must not be required to accept or reject a scan.

**Rationale:** Gate admission feedback must be immediate and reliable. Kafka is appropriate for asynchronous reporting, not the source of truth for the scan result.

**Alternatives considered:** Queuing all scans to Kafka first would introduce uncertainty at the gate and would not fit the PostgreSQL authority requirement.

### Decision 6: Rate-limit sync by device and staff user without blocking local scan capture

Redis-backed rate limiting will apply to preload and sync endpoints, including limits by authenticated staff user, source device ID, and concert ID. If the backend returns `429`, the mobile app keeps pending scan logs locally and retries later.

**Rationale:** Sync endpoints need protection during event peaks, but local scanning must continue during backend throttling or network instability.

**Alternatives considered:** No rate limiting could overload the backend when many devices reconnect together. Blocking local scans on rate-limit failures would harm gate operations.

## Data Model Notes

The feature uses existing global tables: `tickets`, `vip_guests`, `check_ins`, `users`, `roles`, `permissions`, `user_roles`, `role_permissions`, and `audit_logs`.

Implementation may add narrowly scoped metadata to support idempotent offline sync:

- `check_ins.local_scan_id` for the mobile-generated scan identifier.
- `check_ins.source_device_id` for the staff device identifier.
- `check_ins.result_code` or equivalent status values for accepted, duplicate, invalid, expired, unauthorized, and conflict outcomes.
- `check_ins.synced_at`, `check_ins.scanned_at`, and `check_ins.sync_status` if not already present in the implementation.
- A unique index on `(source_device_id, local_scan_id)` for idempotent retry handling.
- Partial unique indexes allowing only one successful check-in per `ticket_id` and per `vip_guest_id`.

If the implementation already has equivalent columns or indexes, reuse them instead of adding duplicates.

## API Shape

- `GET /check-in/assignments`: returns concerts or gates available to the authenticated staff user.
- `GET /check-in/events/:concertId/preload`: returns assignment-scoped validation snapshot data for offline use.
- `POST /check-in/events/:concertId/sync`: accepts a batch of local scan logs and returns one outcome per scan.

All endpoints require authenticated Check-in Staff permissions and assignment checks for the target concert or gate. Backend domain services must repeat these checks before returning validation data or writing check-in records.

## Risks / Trade-offs

- Offline snapshots can become stale after preload -> The app must display snapshot age and refresh when online; backend sync still makes final validity decisions.
- Two devices can accept the same QR code offline -> PostgreSQL unique successful-check-in constraints and first-valid-sync-wins conflict handling resolve the final state.
- A device can fail after scanning but before sync -> The scan is written to Room before the app reports the local result, and WorkManager retries pending uploads.
- Reconnection storms can overload sync APIs -> Redis rate limiting returns retryable responses while local scan logs remain pending.
- Lost or compromised staff devices expose cached event validation data -> Preload is assignment-scoped, requires authenticated staff access, and should store only validation-minimal payloads rather than full audience order data.
- Kafka outage could affect downstream analytics -> The synchronous sync API must not depend on Kafka success for check-in acceptance.

## Migration Plan

1. Add backend check-in module endpoints, DTOs, services, guards, and persistence changes.
2. Add database migration for any missing check-in sync metadata and indexes.
3. Add seed data for local Check-in Staff role permissions, staff assignment demo data, issued tickets, VIP guests, and assigned devices where needed.
4. Add Android mobile app foundations for Room entities, QR scan flow, preload, local validation, and WorkManager sync.
5. Run the local Docker Compose stack and validate online, offline, retry, duplicate, and conflict scenarios.

Rollback is limited to the local prototype: revert the mobile feature and backend check-in module changes, then reverse the database migration if applied. Existing ticket issuance, payment, concert browsing, and organizer flows must remain unaffected.

## Open Questions

- None for the proposal scope. Exact route names and column names can be adjusted during implementation to match the existing codebase, as long as the specified behavior remains intact.
