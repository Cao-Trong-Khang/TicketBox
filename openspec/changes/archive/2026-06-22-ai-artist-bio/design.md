## Context

TicketBox already uses a NestJS modular monolith, PostgreSQL as the source of truth, Kafka for asynchronous work, Redis cache-aside for public concert detail, and JWT/RBAC with service-level ownership checks. This change adds the globally planned AI artist-bio workflow: an Organizer uploads a PDF press kit, the API stores it and enqueues work, a background worker extracts and cleans text and calls an external AI model, and Audience users see only completed content.

The change crosses the Web Application, Backend API, Background Worker, PostgreSQL, Kafka, Redis, local Docker Compose, MinIO, and an external AI model. Check-in Mobile App, payment providers, email, and sponsor CSV processing are not involved.

## Goals / Non-Goals

**Goals:**

- Return an acknowledged upload within the request path without waiting for PDF extraction or AI generation.
- Enforce PDF/size validation, Organizer role/permission, and concert ownership at both controller/guard and domain-service layers.
- Persist observable, recoverable document and biography states in PostgreSQL.
- Handle Kafka at-least-once delivery and distinguish permanent extraction failures from retryable storage and AI failures.
- Let Organizers inspect status, regenerate from an existing upload, and manually edit completed output.
- Expose only the latest completed biography through public concert detail and keep Redis-cached details coherent.
- Keep AI provider choice behind an adapter and support a deterministic local mock.

**Non-Goals:**

- OCR for scanned/image-only PDFs, non-PDF document formats, or press-kit authoring.
- Synchronous generation, streaming progress, WebSocket/SSE updates, or automatic publication approval workflows.
- Production object-storage provisioning, cloud deployment, model training, or provider billing controls.
- Changes to checkout, payment, notifications, VIP import, or check-in behavior.

## Decisions

### 1. Add a bounded artist-bio module and a dedicated worker entry point

The Backend API will expose an `artist-documents` module with controller, application service, storage port, Kafka publisher port, and persistence through Prisma. A separate NestJS worker process will consume `ai.bio.requested` and use PDF extraction and AI-provider ports. This preserves the existing layered modular-monolith and event-driven architecture while isolating slow work from API latency.

Alternative considered: execute extraction and generation in the upload request. Rejected because external storage and AI latency/failure would block the organizer request and contradict the asynchronous global design.

### 2. Store binaries in MinIO and authoritative state in PostgreSQL

MinIO will run in local Docker Compose with a private bucket. Object keys will be generated by the server (for example `artist-documents/<concertId>/<documentId>.pdf`), never accepted from the client, and accessed only through the backend. PostgreSQL will add:

- `artist_documents`: UUID id, concert FK, file name, storage key, status enum/string, nullable extracted text, uploaded/created timestamps.
- `ai_artist_bios`: UUID id, unique document FK, concert FK, status enum/string, nullable generated bio and failure reason, nullable generated timestamp, created timestamp.

Indexes cover `artist_documents(concert_id, created_at)` and `ai_artist_bios(concert_id, status, created_at)`. Foreign keys cascade when a concert/document is intentionally removed. A biography row is created/upserted as `generating` when processing begins so extraction and AI failures have a canonical `failure_reason`; `artist_documents.status` remains the coarse pipeline status.

Alternative considered: store PDFs or generated state in Redis. Rejected because Redis is not authoritative and large binary values would undermine cache usage.

### 3. Validate before storage and acknowledge only after Kafka accepts the event

The upload controller uses multipart field `file`, a 10 MB limit, PDF extension plus MIME type validation, and a PDF signature check before object upload. The service verifies authenticated Organizer permission and `concert.organizer_id`, creates the document as `uploaded`, publishes `{ document_id, concert_id, storage_key, attempt: 1 }` keyed by document id, and returns HTTP 202 only after broker acknowledgement.

If object upload fails, no document is created. If database creation fails, the object is deleted best-effort. If Kafka publication fails, the document remains `uploaded`, the API returns the standard 503 error envelope, and a retry/reconciliation path may safely republish the same document-id event; consumers are idempotent. This avoids falsely returning 202 while acknowledging that MinIO, PostgreSQL, and Kafka cannot share one atomic transaction.

Alternative considered: add a transactional outbox. It offers stronger atomic delivery but adds a new shared infrastructure pattern and table beyond the requested prototype; it can replace reconciliation later if delivery guarantees require it.

### 4. Make processing idempotent under Kafka at-least-once delivery

The worker loads the document by both document and concert ids and treats `done` as terminal. State transitions use conditional updates, and the biography record is upserted by unique `document_id`; duplicate events cannot create multiple bios for one document. The consumer commits the Kafka offset only after a terminal `done` or non-retryable `failed` state is durable. A crash before commit causes redelivery. An AI call can still be repeated after a crash between provider success and database commit, but the database result remains single-valued.

The pipeline is:

1. `uploaded -> extracting`; download the object from MinIO.
2. Extract and normalize whitespace/control characters; fail permanently when cleaned text is under 50 characters.
3. Save full cleaned text and transition `extracting -> extracted -> generating`.
4. Send a fixed biography prompt plus at most the first 4,000 cleaned characters through the configured AI adapter.
5. Upsert the bio as `done` with `generated_at`, mark the document `done`, and invalidate concert-detail cache.

MinIO timeouts retry up to three attempts with exponential backoff. AI timeouts retry up to two attempts. AI rate limits wait at least 60 seconds before retrying. Exhaustion marks both records failed with a sanitized reason. Non-text PDFs fail without retry with `Could not extract text. Please upload a text-based PDF.`

Alternative considered: one Kafka topic per pipeline stage. Rejected because the workflow is small and sequential; one keyed job plus durable state is simpler for this modular monolith.

### 5. Regeneration creates a new immutable processing attempt

`POST .../:documentId/regenerate` verifies ownership, copies the original document's `file_name` and `storage_key` into a new `artist_documents` row with a new UUID and `uploaded` status, publishes a new event, and returns the new id. The stored PDF is reused rather than uploaded again. Historical attempts remain available and the newest completed bio wins by `ai_artist_bios.created_at`.

Alternative considered: reset the original row. Rejected because it destroys failure/history information and makes duplicate-event handling ambiguous.

### 6. Manual edits update the selected document bio and become public immediately

The manual-edit endpoint requires a non-empty bounded biography string and an existing biography for that owned document. It updates `generated_bio`, status `done`, and `generated_at`, keeps the document `done`, records the sensitive action in the existing audit mechanism where available, and invalidates the public concert-detail cache. A missing/generated-in-progress bio returns 409 rather than silently creating an unrelated record.

### 7. Public detail uses the latest completed bio and omits every other state

The public concert-detail query left-joins/selects the newest `ai_artist_bios` row for the concert with status `done`, ordered by `created_at DESC`. The serializer adds `artist_bio` only when a completed non-empty value exists; absence, processing, and failure produce no property. Completion and manual edits invalidate `concerts:detail:<concertId>` so Redis does not keep stale biography content.

### 8. Admin UI polls a stable status resource

The Organizer Web Application validates `.pdf` and 10 MB before upload, then polls document detail every 3–5 seconds while status is non-terminal. Polling stops on `done`, `failed`, unmount, or replacement by a new attempt. It shows generated output, sanitized failure reason, Regenerate, and Edit Manually actions. Server validation remains authoritative.

### 9. Standardize security and error boundaries

All admin endpoints require Bearer authentication, Organizer role/permission, and a repeated service-level ownership query. The service also checks that `documentId` belongs to `concertId` to prevent cross-concert references. Storage keys, raw provider errors, prompts, and credentials are never returned. New endpoint errors use `{ error, message, status_code }`; public browsing remains available when MinIO, Kafka, or the AI provider is unavailable.

## Risks / Trade-offs

- [PDF parsing can consume CPU or memory] → Enforce 10 MB before parsing, process only in the worker, configure extraction limits/timeouts, and never parse in the API process.
- [At-least-once delivery may repeat an AI request] → Key events by document id, use conditional states and unique upsert, and accept rare repeated provider calls after a crash.
- [Database/object/broker writes are not atomic] → Return 202 only after broker acknowledgement, use best-effort cleanup and idempotent reconciliation; consider an outbox if prototype guarantees prove insufficient.
- [Provider output can be inaccurate or unsafe] → Treat it as organizer-reviewable content, support manual edit, use a constrained prompt, and expose nothing until status is done.
- [Polling creates repeated reads] → Poll only active jobs at 3–5 second intervals, stop at terminal states, and index document lookups.
- [Reusing one MinIO object across regeneration attempts complicates deletion] → Do not delete an object while any document row references its storage key; lifecycle cleanup is deferred.
- [Public cache can show stale content] → Invalidate concert detail on completion/edit and retain normal cache TTL as fallback.

## Migration Plan

1. Add MinIO service, private bucket initialization, environment variables, and local mock-AI configuration.
2. Apply additive Prisma migration for both tables, enums/constraints, foreign keys, unique document-bio mapping, and indexes.
3. Deploy storage/AI adapters, artist-document API module, Kafka publisher, and worker consumer with the feature UI initially hidden.
4. Start the worker, verify MinIO/Kafka connectivity and a mock end-to-end job, then enable Organizer UI and public optional field.
5. Warm or invalidate existing concert-detail cache entries.

Rollback disables uploads and the worker first, removes the optional public/UI integration, and leaves additive tables and stored objects intact for data safety. The schema can be removed only after confirming no retained documents/bios are needed.

## Open Questions

- Which AI provider is the default beyond local development, and what biography length/tone policy should its adapter enforce?
- Should a future production version replace upload reconciliation with a PostgreSQL transactional outbox?
- What retention period should apply to failed/superseded PDFs and extracted press-kit text, which may contain sensitive material?
